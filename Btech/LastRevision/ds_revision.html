Foundations
U1 · 01
What is a Data Structure?
A data structure is a systematic way of organising, storing, and managing data in a computer so that it can be accessed and modified efficiently.
  • Data: raw facts without context
  • Information: processed, meaningful data
  • Organisation: how elements are arranged and related
  • Operations: insertion, deletion, traversal, search, sort, merge
  • Historical note: The field formalised in the 1960s; Donald Knuth's TAOCP (1968) was foundational
FundamentalTAOCPKnuth

Abstract Data Types (ADTs)
An ADT is a mathematical model for a data type defined by its behaviour (semantics) from the user's perspective — specifying what operations do, not how they are implemented.
Components of an ADT
A set of values (domain) + a set of operations with defined behaviour + axioms/postconditions
Why ADTs?
Abstraction separates interface from implementation — code can change internals without breaking callers (information hiding)

Time & Space Complexity
U1 · 04
Asymptotic Notation
Asymptotic analysis describes the limiting behaviour of an algorithm's resource usage as input size n grows to infinity — independent of hardware constants.
O (Big-O) — Upper Bound
Worst-case guarantee. f(n) = O(g(n)) if ∃ c, n₀ such that f(n) ≤ c·g(n) ∀ n ≥ n₀
Ω (Omega) — Lower Bound
Best-case guarantee. f(n) = Ω(g(n)) if ∃ c, n₀ such that f(n) ≥ c·g(n) ∀ n ≥ n₀
Θ (Theta) — Tight Bound
Both O and Ω hold simultaneously — the algorithm is exactly that growth class
Big-OAnalysisAsymptotic

U1 · 06
Space Complexity
Space complexity measures the total memory an algorithm uses relative to input size — including auxiliary space (extra space beyond input storage).
  • In-place algorithm: O(1) auxiliary space (e.g. bubble sort)
  • Out-of-place: O(n) auxiliary space (e.g. merge sort needing temp array)
  • Stack frames: Recursive algorithms consume stack space proportional to recursion depth
  • Trade-off: Time-space trade-off — faster algorithms often need more memory (e.g. hash tables)
SpaceIn-placeTrade-off
U1 · 07
Best, Average & Worst Cases
An algorithm's complexity is analysed across three scenarios that capture different input distributions.
Best Case
Minimum steps on the most favourable input (e.g. target is first element in search)
Average Case
Expected steps over all possible inputs (often requires probabilistic analysis)
Worst Case
Maximum steps on the most unfavourable input — the most commonly reported measure
Searching Techniques
U1 · 08
Linear Search
Sequentially checks each element until the target is found or the list is exhausted. No pre-condition on data order.
  • Algorithm: Start at index 0, compare each element to target, return index if match or -1 if exhausted
  • Works on: Sorted and unsorted arrays, linked lists
  • Best case: O(1) — target at first position
  • Average case: O(n/2) ≈ O(n)
  • Worst case: O(n) — target at last or absent
  • Space: O(1) — no extra memory
  • Limitation: Inefficient for large datasets; O(n) even with sorted data
for i in range(len(arr)):
  if arr[i] == target: return i
return -1
O(n)Unsorted OKSequential
U1 · 09
Binary Search
Efficiently searches a sorted array by repeatedly halving the search interval — comparing the target with the middle element and discarding half the range.
  • Pre-condition: Array must be sorted in ascending order
  • Best case: O(1) — target is the middle element
  • Average & Worst: O(log₂ n) — each step halves the search space
  • Space: O(1) iterative; O(log n) recursive (call stack)
  • Insight: 1 billion elements require at most 30 comparisons
Algorithm Steps
low=0, high=n-1 → mid=(low+high)//2 → if arr[mid]==target return mid → if target < arr[mid], high=mid-1 → else low=mid+1 → repeat until low>high
O(log n)Sorted RequiredDivide & Conquer

Sorting Techniques
U1 · 11
Bubble Sort
Repeatedly steps through the list, compares adjacent elements and swaps them if they are in the wrong order — "bubbling" the largest unsorted element to the end each pass.
  • Invented: 1956; popularised by Knuth in TAOCP
  • Best case: O(n) with optimised flag (already sorted)
  • Average & Worst: O(n²) — nested loops
  • Space: O(1) — in-place
  • Stable: Yes — equal elements maintain relative order
  • Key insight: After k passes, last k elements are in final position
  • Limitation: Extremely slow in practice; worst practical sorting algorithm for large n
Pass 1Compare pairsLargest bubbles to end
Pass 2Shrink range2nd largest in place
…n-1 passesArray sorted
O(n²)StableIn-place
U1 · 12
Selection Sort
Finds the minimum (or maximum) element from the unsorted portion and places it at the beginning — "selecting" the correct element for each position.
  • Best, Average & Worst: All O(n²) — always scans full unsorted portion
  • Space: O(1) — in-place
  • Stable: No by default (swapping can change relative order of equals)
  • Advantage over Bubble: Fewer swaps — at most n-1 swaps total (useful when writes are expensive)
  • Key insight: After k passes, first k elements are in final sorted position
Algorithm Skeleton
For i=0 to n-2: find min in arr[i..n-1] → swap arr[i] with min → repeat
O(n²)Not StableMin Swaps
U1 · 13
Insertion Sort
Builds the sorted array one element at a time — each new element is inserted into its correct position among previously sorted elements, like sorting playing cards in hand.
  • Best case: O(n) — already sorted input (only 1 comparison per element)
  • Average & Worst: O(n²)
  • Space: O(1) — in-place
  • Stable: Yes
  • Adaptive: Yes — efficient on nearly-sorted data
  • Preferred: For small arrays (< 20 elements); used as base case in Timsort (Python's built-in)
  • Online: Can sort a stream — doesn't need all elements upfront
O(n²)StableAdaptiveOnline

Unit II
Linked Lists

Dynamic node-based structures — singly, doubly, circular — with their operations, comparisons, and applications.

Fundamentals
U2 · 01
What is a Linked List?
A linked list is a linear data structure in which elements (nodes) are stored at non-contiguous memory locations, connected via pointers.
  • Node: The fundamental unit — contains data field + pointer field(s)
  • Head: Pointer to the first node; entry point to the list
  • Tail: Last node; its next pointer is NULL
  • Dynamic size: Grows/shrinks at runtime — no pre-allocation needed
  • Invented by: Allen Newell, Cliff Shaw, Herbert Simon (1956) for the IPL language
  • No random access: Must traverse from head — O(n) to reach index i
DynamicPointer-basedNon-contiguous
U2 · 02
Node Structure
Each node is a self-referential structure containing data and one or more links to adjacent nodes.
// Singly linked node
struct Node {
  int data;
  struct Node* next;
};

// Doubly linked node
struct DNode {
  int data;
  struct DNode *prev, *next;
};
Self-referentialStruct
U2 · 03
Types of Linked Lists
Singly Linked List (SLL)
Each node has ONE pointer (next). Traversal only in forward direction. Simplest form. Memory: O(n) for data + n pointers.
Doubly Linked List (DLL)
Each node has TWO pointers (prev + next). Bidirectional traversal. Easier deletion (no need for previous node pointer). Extra memory per node.
Circular Singly Linked List
Last node's next points back to head. Useful for round-robin scheduling. No NULL in the list — requires care to detect end.
Circular Doubly Linked List
DLL where tail's next = head AND head's prev = tail. Most flexible; used in OS (ready queue, LRU cache).
Header/Sentinel List
A dummy head node (sentinel) simplifies edge cases in insertion/deletion — no special handling for empty list or first node.
Skip List
(Beyond syllabus) Multi-layer linked list enabling O(log n) search — probabilistic alternative to balanced trees. Used in Redis.
Singly Linked List Operations
U2 · 04
SLL — Insertion
Inserting a new node by adjusting pointer links — O(1) if position known, O(n) to find position.
At Beginning (Prepend)
newNode→next = head; head = newNode. O(1). New node becomes the head.
At End (Append)
Traverse to last node (O(n)), then lastNode→next = newNode. O(n) without tail pointer; O(1) with tail pointer.
At Position k
Traverse to (k-1)th node, adjust pointers: newNode→next = prev→next; prev→next = newNode. O(k).
After a Given Node
newNode→next = givenNode→next; givenNode→next = newNode. O(1) if given node reference is known.
O(1) headO(n) end
U2 · 05
SLL — Deletion & Traversal
Removing a node by relinking pointers so the deleted node is bypassed; traversal visits each node once.
  • Delete head: head = head→next. O(1)
  • Delete tail: Traverse to second-last (O(n)), set →next = NULL
  • Delete by value: Find node with value, keep reference to previous node, bypass it
  • Delete by position: Traverse to (k-1)th node, set prev→next = prev→next→next
  • Traversal: curr = head; while curr ≠ NULL: process(curr); curr = curr→next. O(n)
  • Search: Traverse until data matches or NULL. O(n)
O(n)Pointer manipulation
U2 · 06
Doubly Linked List — Operations
DLL operations are similar to SLL but involve setting both prev and next pointers — more pointer updates, but allows backward traversal and easier deletion.
  • Insert at beginning: newNode→next = head; head→prev = newNode; head = newNode
  • Insert at end: traverse to tail; tail→next = newNode; newNode→prev = tail
  • Delete a node: No need to track previous — use node→prev directly: node→prev→next = node→next; node→next→prev = node→prev. O(1) given node reference!
  • Backward traversal: Start from tail, follow prev pointers
  • Advantage: O(1) deletion with just a node reference (SLL needs O(n))
  • Disadvantage: Extra memory per node; more pointer updates required
BidirectionalO(1) delete
U2 · 07
Circular Linked List — Operations
In a circular list, the last node loops back to the first. Traversal must detect when we've returned to the starting point to avoid infinite loops.
  • Traversal guard: Stop when curr→next == head (not NULL)
  • Insert at beginning: Also update tail→next to point to new head
  • Advantage: Can reach any node from any starting point — natural for cyclic processes
  • Josephus Problem: Classic algorithm using circular linked list — n people in circle, every k-th eliminated
  • Application: CPU round-robin scheduling, multiplayer turn-based games, circular buffers
No NULLRound-robinCyclic

Unit III
Stacks

The Last-In First-Out (LIFO) principle — from array and linked-list implementations to expression evaluation and backtracking.

Stack Fundamentals
U3 · 01
Stack — Definition & Properties
A stack is a linear data structure that follows the Last In, First Out (LIFO) principle — the element inserted last is the first to be removed.
  • Top: The pointer/index to the most recently inserted element
  • LIFO: Like a stack of plates — you remove from the top
  • Restricted access: Insertions and deletions happen only at the top
  • Analogies: Browser back button, call stack, undo history, coin stack
  • Historical: Formalised by F.L. Bauer and K. Samelson in 1957 for expression evaluation in compilers
LIFOTop pointerRestricted
U3 · 02
Stack Operations
The Stack ADT defines five primary operations — all O(1) time complexity.
push(element)
Add element to top. If top == max-1 → Stack Overflow error. O(1).
pop()
Remove and return top element. If stack empty → Stack Underflow error. O(1).
peek() / top()
Return top element without removing it. O(1).
isEmpty()
Returns true if stack has no elements (top == -1 for array implementation). O(1).
isFull()
Only for array-based stacks — returns true if top == max-1. O(1).
Implementations
U3 · 03
Stack Using Array
Use an array with an integer top (initially -1) as index of the most recent element.
int stack[MAX], top = -1;

void push(int x) {
  if (top == MAX-1) overflow();
  stack[++top] = x;
}
int pop() {
  if (top == -1) underflow();
  return stack[top--];
}
  • Advantage: Simple, cache-friendly, O(1) all operations
  • Disadvantage: Fixed size — overflow possible; wasted memory if underutilised
  • Size: Must be declared upfront (static) or use dynamic array
O(1)Fixed sizeCache friendly
U3 · 04
Stack Using Linked List
The head of the linked list serves as the top of the stack — push prepends, pop removes head. No overflow (limited only by heap memory).
void push(int x) {
  Node* n = new Node(x);
  n→next = top; top = n;
}
int pop() {
  int val = top→data;
  Node* tmp = top;
  top = top→next;
  delete tmp; return val;
}
  • Advantage: Dynamic size — no overflow; no wasted space
  • Disadvantage: Extra pointer overhead per node; dynamic allocation is slower than array access
DynamicNo overflowPointer overhead

Mathematical expressions can be written in three notations — stacks are key to converting and evaluating them.

U3 · 10
Function Call Stack (Activation Records)
The program call stack is the most critical real-world use of the stack ADT — tracking function calls, local variables, and return addresses.
  • Stack frame: Created on each function call — contains local variables, parameters, return address
  • LIFO property: When function returns, its frame is popped — control returns to caller
  • Recursion: Each recursive call adds a new frame; deep recursion can cause Stack Overflow
  • Tail call optimisation: Some compilers replace tail-recursive calls to avoid adding frames
  • Stack size: Typically 1–8 MB per thread; configurable in OS

Unit IV
Queues & Deques

FIFO discipline — ordinary queues, circular queues, priority queues, and the versatile double-ended deque.

Queue Fundamentals
U4 · 01
Queue — Definition & Properties
A queue is a linear data structure that follows First In, First Out (FIFO) — the element inserted first is removed first, like a line of customers.
  • Front/Head: Pointer to the element to be dequeued next
  • Rear/Tail: Pointer to the position where the next element will be inserted
  • FIFO: Contrast with Stack (LIFO) — completely opposite removal order
  • Real-world analogy: Ticket queue, print spooler, hospital waiting list
  • Origin: Formalised in early 1960s for OS process scheduling
FIFOFront + RearScheduling
U4 · 02
Queue Operations
The Queue ADT defines four core operations, all O(1).
enqueue(element)
Insert element at rear. If rear == MAX-1 → overflow. O(1).
dequeue()
Remove and return element from front. If front > rear or empty → underflow. O(1).
peek() / front()
Return front element without removing. O(1).
isEmpty()
Returns true if front > rear (array) or head == NULL (linked list). O(1).
Queue Implementations & Variants
U4 · 03
Queue Using Array — Linear & Circular
A linear array queue wastes space as front advances — the circular queue fixes this by treating the array as a ring.
  • Linear array problem: After dequeues, front advances but vacated slots can't be reused — false "overflow"
  • Circular queue: rear = (rear + 1) % MAX; front = (front + 1) % MAX
  • Full condition: (rear + 1) % MAX == front
  • Empty condition: front == rear (or use a count variable)
  • Effective capacity: MAX - 1 slots (one slot wasted to distinguish full from empty)
  • Also called: Ring buffer, circular buffer — used in OS I/O buffers, audio processing
Ring bufferModulo arithmeticO(1)
U4 · 04
Queue Using Linked List
Maintain front and rear pointers — enqueue at rear, dequeue at front. No size limit, no wasted space.

else front = n;
rear = n;

// Dequeue

Node* tmp = front;
front = front→next;
if (!front) rear = NULL;
delete tmp;

Types of Queues
Several specialised queue variants extend the basic FIFO behaviour for specific use cases.
Simple (Linear) Queue
Basic FIFO. Enqueue at rear, dequeue at front. Fixed size or dynamic.
Circular Queue
Array treated as circular ring. Eliminates the false overflow problem. Used in ring buffers.
Priority Queue
Elements dequeued in order of priority, not insertion order. Min-heap or Max-heap implementation. Used in Dijkstra's algorithm, OS scheduling.
Double-Ended Queue (Deque)
Insert and delete at both ends. Generalises both stack and queue. Covered in detail below.

Applications of Queues
Queues model real-world waiting and scheduling scenarios across systems programming and algorithms.
BFS (Graph Traversal)
Breadth-First Search explores nodes level by level using a queue. Finds shortest path in unweighted graphs.

Deque (Double-Ended Queue)

Deque — Definition & Operations
A Deque (Double-Ended Queue) allows insertion and deletion at both front and rear — a generalisation of both Stack and Queue.
insertFront(x)
Add element to the front. O(1) with DLL or circular array.
insertRear(x)
Add element to the rear. Same as regular enqueue.
deleteFront()
Remove from front. Same as regular dequeue.
deleteRear()
Remove from rear. Like stack pop. Requires DLL or circular array for O(1).
getFront() / getRear()
Peek at front or rear without removal. O(1).
Both endsStack + QueueO(1)
U4 · 08
Deque Variants & Applications
Restricted deques limit operations on one end; real-world deques power sliding window algorithms and palindrome checking.
Input-Restricted Deque
Insertion only at rear; deletion at both ends. Useful when input is always appended but removal is flexible.
Output-Restricted Deque
Deletion only at front; insertion at both ends. When removal order must be FIFO but input can be front-inserted.
Key Applications
  • Sliding window maximum (O(n) algorithm using deque)
  • Palindrome check: Add chars to deque, compare front and rear simultaneously
  • Undo/Redo: Undo = deleteFront, Redo = insertFront
  • Work-stealing schedulers (e.g. Java ForkJoinPool): Threads take tasks from their own deque's rear; steal from others' front

Unit V
Trees & Hashing

Hierarchical data with BST operations, and the mathematical magic of hash functions — from collisions to caching.

Tree Fundamentals
U5 · 01
Tree — Definition & Terminology
A tree is a non-linear hierarchical data structure consisting of nodes connected by edges, with one special root node at the top and no cycles.
Root
The topmost node — has no parent. Entry point to the tree.
Parent / Child
A node directly above is parent; nodes directly below are children.
Leaf (External) node
A node with no children. Also called terminal node.
Internal node
Any non-leaf node — has at least one child.
Height
Number of edges on longest path from root to a leaf. Empty tree: -1; single node: 0.
Depth / Level
Number of edges from root to a given node. Root is at depth 0.
Degree
Number of children of a node. Degree of tree = maximum node degree.
Subtree
Any node + all its descendants forms a subtree.
HierarchicalAcyclic
U5 · 02
Binary Tree — Types
A binary tree is a tree where each node has at most two children — left child and right child.
Full (Strictly) Binary Tree
Every node has 0 or 2 children — no node has exactly 1 child.
Complete Binary Tree
All levels fully filled except possibly the last, which is filled left to right. Max-heaps and min-heaps are complete binary trees.
Perfect Binary Tree
All internal nodes have 2 children AND all leaves are at the same level. Node count = 2^(h+1) - 1.
Degenerate (Skewed) Tree
Each parent has only one child — essentially a linked list. Worst case for BST (O(n) all operations).
Balanced Binary Tree
Height difference between left and right subtrees ≤ 1 for every node. Guarantees O(log n) operations. AVL, Red-Black trees are self-balancing.

BST — Definition & BST Property
A Binary Search Tree is a binary tree where every node's left subtree contains only values less than the node, and every right subtree contains only values greater than the node.
  • BST Property: For every node N: all keys in N.left < N.key < all keys in N.right
  • Inorder traversal of a BST always produces a sorted sequence
  • Average case: O(log n) for search, insert, delete — when tree is balanced
  • Worst case: O(n) — when tree degenerates into a linked list (sorted insertions)
  • Historical: Binary search trees formalised by P.F. Windley and others around 1960
  • No duplicate keys: Standard BSTs don't allow duplicate keys (handled by convention: go left or right)

BST — Search
Search exploits the BST property to eliminate half the remaining tree at each step — similar to binary search on arrays.
search(root, key):
  if root is NULL → return NULL
  if key == root.data → return root
  if key < root.data:
    return search(root.left, key)
  else:
    return search(root.right, key)
  • Best case: O(1) — key is at root
  • Average: O(log n) for balanced BST
  • Worst: O(n) for degenerate tree
  • Key insight: At each node, we definitively eliminate one subtree
O(log n)Recursive
U5 · 06
BST — Deletion
Deletion has three cases depending on the target node's children — the trickiest case uses the inorder successor (or predecessor).
Case 1: Leaf Node
Simply remove the node. Set parent's child pointer to NULL. O(log n) to find.
Case 2: One Child
Bypass the node — link parent directly to the node's only child. The child subtree slides up.
Case 3: Two Children
Find the inorder successor (smallest in right subtree) OR inorder predecessor (largest in left). Copy its value to deleted node. Delete the successor/predecessor (which has at most one child).
Inorder Successor
The leftmost node in the right subtree — the next larger value in sorted order. Always has at most a right child.
3 CasesInorder SuccessorO(log n)
U5 · 07
BST — Tree Traversals
Tree traversal visits every node exactly once in a defined order. Three primary depth-first traversals + one breadth-first traversal.
Inorder (LNR)
Left → Node → Right. Produces sorted sequence for BST. Used for: sorted output, converting BST to sorted array.
Preorder (NLR)
Node → Left → Right. Used for: copying a tree, serialising tree structure, expression tree prefix notation.
Postorder (LRN)
Left → Right → Node. Used for: deleting a tree (children before parent), postfix expression evaluation, computing directory sizes.
Level-order (BFS)
Visit level by level, left to right. Uses a Queue. Used for: finding shortest path, printing tree by level.

Hashing — Introduction & Concepts
Hashing is a technique of converting a key into an index (hash) for direct storage/retrieval in a fixed-size table — achieving O(1) average-case access.
  • Hash function h(k): Maps key k → index in range [0, m-1] where m is table size
  • Hash table: Array indexed by hash values
  • Perfect hashing: No two keys map to same index (rarely achievable for arbitrary keys)
  • Load factor α: α = n/m where n = number of keys, m = table size. Performance degrades as α approaches 1
  • Origin: Developed by H.P. Luhn (IBM) around 1953; formalised by Dumey in 1956
  • Amortised O(1): Average case for insert, search, delete — worst case O(n) if all keys collide
O(1) averageKey-to-indexLoad factor
U5 · 09
Hash Functions
A good hash function distributes keys uniformly, is fast to compute, and minimises collisions.
Division Method
h(k) = k mod m. Simple and fast. Choose m as a prime not close to a power of 2 for best distribution.
Multiplication Method
h(k) = ⌊m · (k · A mod 1)⌋ where 0 < A < 1. Knuth suggests A ≈ (√5 - 1)/2 ≈ 0.618 (golden ratio). Works for any m.
Universal Hashing
Select h randomly from a family of functions at runtime — prevents adversarial inputs from causing worst-case collisions.
Folding Method
Split key into parts, add them together, take mod. e.g. key=12345678 → 12+34+56+78=180 → 180 mod m.
Properties of a Good Hash Function
Deterministic · Uniform distribution · Fast computation · Avalanche effect (small input change → large output change)
DivisionMultiplicationUniversal
U5 · 10
Collision & Chaining
A collision occurs when two different keys hash to the same index. Chaining (separate chaining) resolves collisions by storing all colliding keys in a linked list at each bucket.
  • Separate Chaining: Each slot of the hash table is a linked list (or dynamic array) of all keys mapping to that index
  • Insertion: Hash key → prepend to list at that index. O(1)
  • Search: Hash key → search list at index. O(1 + α) average where α = load factor
  • Load factor: Can exceed 1 with chaining (unlike open addressing)
  • Advantage: Simple; table never "fills up"; handles high load factors gracefully
  • Disadvantage: Extra memory for pointers; poor cache performance (pointer chasing)
  • Real world: Java HashMap, Python dict (pre-3.6) used chaining

U5 · 11
Open Addressing
When a collision occurs, open addressing finds the next available slot in the table itself — all entries stored within the table array (no extra linked lists).
Linear Probing
h(k, i) = (h(k) + i) mod m. Try next slot, then next, etc. Simple but causes primary clustering — long runs of occupied slots accumulate, degrading performance.
Quadratic Probing
h(k, i) = (h(k) + c₁i + c₂i²) mod m. Reduces primary clustering but causes secondary clustering — keys with same initial hash follow same probe sequence.
Double Hashing
h(k, i) = (h₁(k) + i·h₂(k)) mod m. Uses a second hash function to determine probe step — virtually eliminates clustering. Best open addressing method.
Deletion Challenge
Deleting a key from open addressing is tricky — must mark as "DELETED" (tombstone), not truly empty, or searches break for subsequent probes.

U5 · 13
Applications of Hashing
Hashing is one of the most widely used data structure techniques in real-world systems.
Caching (Memoisation)
Store previously computed results in a hash map — O(1) lookup. Used in dynamic programming, browser caches (URL → content).
Unique ID / Deduplication
Hash set tracks seen elements — detect duplicates in O(n) instead of O(n²). Used in finding duplicate files, IPs, records.
Database Indexing
Hash indexes in databases (e.g. PostgreSQL) allow O(1) equality queries — faster than B-tree for exact lookups.
Cryptographic Hashing
SHA-256, MD5 — one-way hash functions for password storage, data integrity. Different design goals from hash tables.
Symbol Tables in Compilers
Variable names → attributes stored in hash map during compilation — fast identifier lookup.
LRU Cache
HashMap + Doubly Linked List together implement O(1) LRU (Least Recently Used) eviction — used in OS page replacement, CDNs.

Array Reversal
Reverse an array in-place using two pointers moving from opposite ends towards the centre.
void reverse(int arr[], int n) {
  int l = 0, r = n - 1;
  while (l < r) {
    swap(arr[l++], arr[r--]);
  }
}
Complexity
Time: O(n/2) = O(n) · Space: O(1) — truly in-place
  • Alternative: Use a stack — push all elements, then pop. O(n) space.
  • Application: Rotating arrays, palindrome check, reversing strings

Lab Tips & Common Mistakes
TIP · 01
Pointer Pitfalls
The most common source of bugs in linked list implementations.
  • NULL dereference: Always check if pointer is NULL before accessing →next or →data
  • Lost nodes: Save next pointer before overwriting it during reversal or deletion
  • Circular infinite loop: In CLL traversal, use do-while or check before visiting, not after
  • Memory leak: In C/C++, always free() / delete deleted nodes
  • Off-by-one: When inserting at position k, stop at (k-1), not k
TIP · 02
Stack / Queue Edge Cases
Always handle boundary conditions before implementing the happy path.
  • Underflow: pop/dequeue on empty structure — add isEmpty() check first
  • Overflow: push/enqueue on full array structure — add isFull() check
  • Single element: After popping last element, ensure top = -1 (stack) or front = rear = -1 (queue)
  • Circular queue distinction: One slot always left empty — effective capacity is MAX-1
  • Postfix operand order: val2 = pop() first, val1 = pop() second — order matters for -, /, ^
TIP · 03
BST & Hashing Tips
Key insights for implementing trees and hash tables correctly.
  • BST deletion two-children: Don't physically move nodes — copy inorder successor's data, then delete the successor node
  • Traversal base case: Always start with: if (root == NULL) return — before any recursion
  • Hash table prime m: Choose table size as prime number for better distribution with division method
  • Tombstones: In open addressing, use a sentinel DELETED marker — don't set to NULL or searches break
  • Load factor: Resize hash table (rehash all keys) when α > 0.7 for performance
